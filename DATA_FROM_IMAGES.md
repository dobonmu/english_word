# 이미지(필기 사진)를 단어/Writing/Speaking 데이터로 변환하는 규칙

사용자가 `.image/` 폴더 아래에 단어장·workbook을 촬영한 사진을 넣으면, 이를 읽고
`words/`, `writing/`, `speaking/` 아래의 `.txt` 데이터 파일로 옮겨 적는 작업이 반복적으로
발생한다. 이 문서는 그 변환 규칙을 정리한 것이다. Claude든 다른 AI든, 이미지만 보고
이 문서를 참고하면 동일한 결과물을 만들 수 있어야 한다.

## 0. 공통 원칙

- **`.image/`는 항상 `.gitignore`에 있어야 하고, 절대 git에 커밋/push하지 않는다.**
  원본 사진은 개인 학습 자료이며 저장소에는 변환된 텍스트만 들어간다.
- 이미지는 보통 여러 장(때로 20장 이상)으로 나뉘어 있고, 파일명 뒤 숫자(001, 002, ...)가
  페이지 순서다. **번호가 중간에 비어있으면(예: 001 다음 004) 파일이 실제로 없는지
  먼저 `ls`로 확인**한다 — 다른 하위 폴더에 흩어져 있는 경우가 있었다(예:
  `.image/words/`와 `.image/writing/`이 별도 폴더였던 사례).
- 이미지를 다 읽기 전에 형식을 넘겨짚지 않는다. 최소 2~3장을 먼저 읽어 표/문서 형식을
  파악한 뒤 나머지를 순서대로 읽는다.
- 표 안의 번호(1, 2, 3...)가 연속되지 않고 중간부터 시작하면(예: 3번부터 시작) 앞부분
  페이지가 빠졌다는 신호다. 사용자에게 재확인하거나 빠진 번호만큼 개수가 줄어든 채로
  진행하지 말고, 반드시 빠진 페이지가 있는지 폴더를 다시 확인한다.
- 사용자가 나중에 이미지를 더 추가하며 "이것도 추가해줘"라고 하면, **기존 데이터를
  갈아엎지 말고 이어서 추가**하는 것이 기본이나, 이미 잘못 매칭된 상태였다면(예: 사용자가
  요구한 정식 단어장과 다른 임의 매칭으로 잘못 만든 경우) 전체를 정식 자료로 교체하는 것이
  맞다. 애매하면 사용자에게 확인한다.

## 1. `words/*.txt` — 단어장 (VOCAB_DATA)

### 원본 이미지 형식
"필수 영단어 N" 같은 표 형식 워크북. 컬럼은 보통 `번호 | 단어 | 의미 | 예문(English / 한국어)`.
한 페이지에 7~10개 단어씩 실려 있고, 표가 세로로 길게 이어진다(사진이 회전되어 있는 경우 많음).
페이지 하단에 사용자가 손글씨로 동의어/문법 메모를 남겨둔 경우가 있다 — 이건 note 필드로
흡수한다.

### 출력 형식
파일명 = 단원 이름 그대로(`1단원.txt`, `2단원.txt`, `필수 영단어 3.txt` 등, 확장자만 `.txt`).
각 줄:

```
단어|뜻|예문(영어)|예문(한글)|메모(선택, 문법/동의어/유의점 등)
```

- **단어**: 표에 적힌 그대로(첫 글자 대문자, 예: `Capacity`). 원문 그대로 옮기고 임의로
  바꾸지 않는다.
- **뜻**: 콤마로 여러 의미가 나열되어 있으면 그대로 콤마로 옮긴다 (`부족한, 결핍된`).
- **예문(영어/한글)**: 표에 있는 문장을 그대로. 줄바꿈된 문장은 공백으로 이어붙인다.
- **메모(note)**: 사용자가 별도로 손글씨 또는 텍스트로 남긴 동의어/반의어/문법 패턴을
  옮긴다. 원본 필기가 축약되어 있으면(예: "cause A to B") 그대로 두거나 약간만 풀어써서
  이해 가능하게 한다. 없으면 필드를 비워둔다(마지막 `|` 뒤를 생략해도 됨).
  - 여러 개념이 섞여 있으면 `/`로 구분한다. 예:
    `신중한: considerate / 의도적인: intended, planned`
  - "감소하다"와 "거절하다"처럼 뜻이 여러 개고 각각 다른 동의어 세트가 있으면
    `감소하다 동의어: ... / 거절하다 동의어: ...` 식으로 뜻별로 라벨을 붙인다.

### 흔한 실수
- 표가 페이지 경계에서 끊겨서 1~2번 항목이 다른 사진에 있는 경우를 놓치기 쉽다.
  마지막에 반드시 "1번부터 N번까지 빠짐없이 있는지" 개수를 세어 검증한다.
- 사용자가 준 번호 매김 목록(동의어 메모)을 단어에 매칭할 때, **반드시 원본 이미지의
  번호와 정확히 일치시킨다**. 짐작으로 임의 매칭하면 나중에 통째로 다시 해야 한다
  (실제로 이런 재작업이 있었다 — 처음에 이미지 없이 뜻만 보고 대표 단어를 임의로
  골랐다가, 사용자가 정식 이미지를 보내줘서 전면 교체한 사례).

### 빌드
`python3 build_words.py`를 실행해야 `js/words_data.js`가 갱신된다. txt만 고치고
빌드를 잊으면 화면에 반영되지 않는다.

## 2. `writing/*.txt` — 문장/이메일 표현 학습 (WRITING_DATA)

### 원본 이미지 형식
편지/이메일/에세이 등 형식화된 글의 문장들이 순서대로 나열되고, 옆에 한글 해석과
"학습 포인트"(문형 패턴, 시작/마무리 관용구, 동의어 대체 표현)가 표로 정리되어 있다.

### 출력 형식
파일명 = 학습 세트 이름(예: `시설개선건의편지.txt`). 각 줄:

```
영어문장|한글해석|학습포인트(선택)
```

- 문장은 원본 그대로, 줄바꿈 없이 한 줄로.
- 학습포인트는 **"라벨: 표현1, 표현2, 표현3" 형태를 기본으로 하고, 여러 그룹이 있으면
  `/`로 구분**한다. 예:
  ```
  다양한(동의어): a wide range of, wide, comprehensive, extensive, broad, a diverse array of, a variety of, an assortment of
  ```
  ```
  인접한: adjacent to / 바로 맞은편: right across
  ```
- **라벨이 없어도 콤마로 개별 표현을 나열하면 앱이 자동으로 각각을 분리해서 보여준다**
  (`js/view_writing.js`의 `parsePoint`). 그래도 가능하면 라벨을 붙이는 게 이해하기
  쉽다 — 라벨 없이 밋밋하게 나열하지 말고 "~라는 조건하에: provided that, providing
  that, ..." 처럼 무엇에 대한 대체 표현인지 앞에 적어준다.
- 발음 표기가 있으면 `표현(한글발음)` 형태로 붙여도 되고, 화면에서 자동으로 괄호 안
  발음은 하이라이트 매칭 시 무시된다(`js/word_mark.js`의 `highlightPointInSentence`가
  `\([^)]*\)`를 제거하고 매칭한다).

### 문장 내 하이라이트 동작 원리 (알아두면 좋음)
"문장 보기" 화면은 학습포인트의 각 표현이 실제 영어 문장 안에 등장하면 자동으로
빨간색(`<mark class="hl-point">`)으로 강조한다. 즉:
- 학습포인트에 적은 표현이 문장에 **정확한 부분 문자열로(대소문자 무시)** 등장해야
  강조된다. 문장에 없는 참고 어휘(예: 문장엔 없지만 관련해서 적어둔 동의어)는 강조되지
  않는다 — 이건 정상 동작이지 버그가 아니다.
- 여러 표현이 겹치면 긴 표현을 우선 매칭한다.

### 빌드
동일하게 `python3 build_words.py` 필요.

## 3. `speaking/*.txt` — 스피킹 Q&A (SPEAKING_DATA)

### 원본 이미지 형식
"Part N. 주제별 공략: OOO" 제목 아래 "01. 소주제" → `Q: 질문(영어)` → `(한글 번역)` →
`Sample Answer:` → 모범답변(영어) → `(한글 번역)` 순서로 반복되는 스피킹 문제집.
답변 문장 안에 색깔로 하이라이트되거나 밑줄 쳐진 대체 가능한 표현들이 있고, 페이지
하단이나 다음 줄에 `>표현(한글 발음)` 형태로 동의어/대체 표현이 정리되어 있다.
가끔 "실전 스피킹 팁" 같은 부가 페이지(정답 문답이 아닌 팁)가 섞여 있는데, 이건
데이터로 옮기지 않는다(연습문제 껍데기만 있고 답이 없는 페이지는 제외).

### 출력 형식
파일명 = Part 이름(예: `Part1_교육과진로.txt`, 번호 접두어를 붙이면 자연스러운 순서로
정렬된다 — `natural_key` 정렬 사용). 각 줄:

```
질문(영어)|질문(한글)|모범답변(영어)|모범답변(한글)|대체가능표현(선택)
```

- **질문(한글)**: 원본에 있는 괄호 안 번역을 그대로 쓴다. "주제 설명: 질문" 처럼
  카테고리명이 섞여 있으면 카테고리명은 빼고 순수 질문 번역만 남긴다.
- **모범답변(영어)**: 답변 문장을 그대로 옮기되, **한글 안내문이나 필기 메모가 답변
  문장 중간에 섞여 들어가지 않도록 주의**한다. (실수 사례: `a_en` 필드에
  `"Keep my eye on the ball"를 사용하면 도움이 됩니다.` 같은 한글 문장이 섞여 들어간
  적이 있다 — 이런 안내문은 반드시 학습포인트(5번째 필드)로 옮기고 답변 본문에서는
  제거한다.)
- **대체가능표현**: writing과 동일한 문법(`라벨: 표현1, 표현2 / 라벨2: 표현3`)을 쓴다.
  하이라이트된 단어 + 괄호 안 발음 표기가 있으면 발음은 살려도 되고 빼도 되지만,
  일관성 있게 처리한다.

### 흔한 실수
- Part 안에 소주제가 2개뿐인 경우와 5개인 경우가 섞여 있을 수 있다(실제로 Part1은
  5문답, Part2~4는 각 2문답이었다). 이미지에 있는 만큼만 옮기고 억지로 개수를 맞추지
  않는다.
- 인쇄된 "최종본" 페이지(손글씨 필기 없는 깨끗한 버전)와 손글씨 필기가 있는 페이지가
  같은 내용을 중복으로 담고 있는 경우가 있다. 중복 페이지는 건너뛰고, 필기가 있는
  페이지 쪽의 대체 표현 정보를 우선 채택한다.

### 빌드
동일하게 `python3 build_words.py` 필요. 빌드 스크립트가 `speaking/*.txt`도 함께
읽어서 `SPEAKING_DATA`를 생성한다.

## 4. 작업 후 항상 확인할 것

1. `python3 build_words.py` 실행 → 콘솔에 출력되는 단원별/세트별 개수가 예상과
   맞는지 확인 (예: "3단원: 50개 단어" 처럼).
2. 로컬 서버(`python3 -m http.server`)로 띄운 뒤 Playwright 등으로 실제 화면에서
   새 데이터가 깨지지 않고 보이는지 확인한다 — 특히 학습포인트 콤마/슬래시 파싱이
   의도대로 그룹화되는지, 문장 안 하이라이트가 과도하거나 전혀 안 되지는 않는지.
3. `.image/`가 `git status`에 잡히지 않는지 확인 후 `./push.sh`로 배포한다.
